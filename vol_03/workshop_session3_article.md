## ブレストメモ

### 第二回からの接続

第二回で「マークアップ（宣言的）とプログラミング（手続き的）は全く別のスキル」と整理した。
第三回ではこの2つが合流する場としてのフレームワークを扱う。

### 伝えたいこと

- マークアップとプログラミングを両方やらないといけなくなった結果、「宣言的に動的UIを書ける」Reactが生まれた
   - 手続き的に書かないと行けなかった自体のJSで作ったUIコード
   - ReactのUIコード
- コンポーネント指向というプログラミング的な考え方
- アクセシビリティのためにも使えるUIフレームワークの存在
   - 予め用意されたマークアップを使える
   - Tailwindはデザイナーの力量が問われる
   - Bootstrapはデザイン要素が不要だがきれいに見せたい場合に有効
- マークアップエンジニアの実際の業務に役立つフレームワークの導入方法

### 宣言的UIフレームワーク（React）が生まれた背景

#### 課題
- jQueryなどで手続き的にDOMを操作するのが主流だったが、アプリが複雑になると管理が困難に

#### Reactの革新（2011年、Facebook）
- **宣言的UI**: DOMを直接操作するのではなく、UIが「どうあるべきか」を宣言する
- **コンポーネントベース**: 再利用可能な小さな部品にUIを分割

#### PHPサーバーサイド開発との類似性
- Reactの開発体験は、PHPでサーバーサイドのコードを書いていたときの感覚に似ている
- **リクエスト・レスポンスの繰り返し**: PHPでは毎回HTMLを生成して返す。Reactも状態が変わるたびにUIを「再描画」する感覚（実際はVirtualDOMによる差分描画）
- **HTMLの中にコードを書く**: PHPではHTMLの中にPHPコードを埋め込む。ReactのJSXはJavascriptの中にHTMLを埋め込む。既視感があった。
- ReactはXHP（FacebookのPHP用HTMLコンポーネントライブラリ）からインスパイアされている

```php
<!-- PHP: HTMLの中にPHPを埋め込む -->
<div class="user">
  <h2><?php echo $user->name; ?></h2>
  <p><?php echo $user->email; ?></p>
</div>
```

```jsx
// React (JSX): HTMLライクな記法の中にJSを埋め込む
<div className="user">
  <h2>{user.name}</h2>
  <p>{user.email}</p>
</div>
```

### ReactとUIフレームワークの相性

- ReactのコンポーネントとUIフレームワーク（Chakra UI, MUI, Radix等）は相性が良い
- コンポーネント内で手続き型（ロジック）と宣言型（JSX）の両方を乗りこなす
- → 第二回で整理した「マークアップとプログラミングの違い」が合流する場所
- Tailwindはスタイリング手法であり、UIコンポーネントを提供するものではないので別カテゴリ

### TwitterがBootstrapを開発した背景

#### 課題（2010年）
- Twitter社内で複数のエンジニアが独自のスタイルで開発し、UIに一貫性がなかった

#### 解決策としてのBootstrap
- ボタン、フォーム、ナビゲーションバーなど、プリセットのUIコンポーネントを提供
- デザイナーがいなくても、それなりに見栄えの良いUIが作れる
- 2011年8月にオープンソース化

### Tailwind CSSが生まれた背景（Bootstrapとは全く別のニーズ）

#### Bootstrapとの根本的な違い
- **Bootstrap**: UIフレームワーク。プリセットのコンポーネント（button, card, navbar等）を提供
- **Tailwind**: ユーティリティファーストフレームワーク。プリセットコンポーネントは持たない
- 解決しようとしている問題が根本的に異なる

#### Tailwindが解決しようとした問題
- Adam Wathanが2017年にKiteTailというサイドプロジェクトを開発中に生まれた
- 「長年メンテナブルに感じるCSSのアプローチ」を模索していた
- 最初はBootstrap的なアプローチ（button, cardなどのコンポーネントクラス）で始めた
- しかし徐々に小さなユーティリティクラス（padding, margin, flexbox等）を追加するように
- 「コンポーネントクラスより、ユーティリティクラスの方が使いやすい」と気づいた

#### CSSを書く苦しみからの解放
- 「クラス名を考えるのがつらい」という開発者の声
- CSSファイルとHTMLファイルを行き来するのが面倒
- 「このスタイルはどこで定義されてる？」問題
- Tailwindは「CSSを書かなくていい」という開発体験を提供

#### 2つのフレームワークの対比

| | Bootstrap | Tailwind |
|---|---|---|
| 生まれた場所 | Twitter社内ツール | 個人の副業プロジェクト |
| 解決する問題 | UIの一貫性・デザイン不要 | CSS設計・クラス命名の苦しみ |
| 提供するもの | プリセットコンポーネント | ユーティリティクラス |
| デザイン | Bootstrapっぽい見た目になる | 自由（デザイナーの力量が問われる） |
| 向いている場面 | 管理画面、MVP、デザイナー不在 | カスタムデザイン、コンポーネント設計 |

### 記事の構成案

1. **各種フレームワークの誕生の経緯**
   - React: 宣言的UIの誕生（jQuery時代の課題→PHPとの類似性）
   - Bootstrap: UIの一貫性問題を解決
   - Tailwind: CSS設計の苦しみを解決

2. **特性**
   - 何を提供するか（コンポーネント / ユーティリティクラス / 宣言的UI）
   - ReactとUIフレームワークの相性

3. **どういう場所に適しているか**
   - Bootstrap: 管理画面、MVP、デザイナー不在
   - Tailwind: カスタムデザイン
   - React + UIフレームワーク: 動的なWebアプリ
   - Astro: Node.jsランタイムで動く静的サイトジェネレータ
     - React/Vue/Svelteなど複数のUIフレームワークを混在可能
     - デフォルトでJavaScriptを出力しない（必要な部分だけ出力）
